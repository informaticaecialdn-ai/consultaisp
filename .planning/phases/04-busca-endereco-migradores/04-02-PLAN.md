<objective>
Create the migrator serial detection service that analyzes CPF consultation results to detect fraud-by-migration patterns.

Purpose: When a provider consults a CPF, detect if the person is a "serial migrator" — someone who accumulates debt at one ISP, gets their contract cancelled, and immediately signs up at another ISP in the same region. This is the most common fraud pattern in regional ISPs.
Output: `server/services/migrator-detection.service.ts` with detectMigrator function ready to be called from the consultas route.
</objective>

    Create server/services/migrator-detection.service.ts with:

    1. Export interface MigratorDetectionInput:
       - cpfCnpj: string
       - consultingProviderId: number (the provider doing the consultation)
       - consultingProviderName: string
       - erpResults: RealtimeQueryResult[] (results from all regional ERPs)
       - recentConsultationsByDistinctProviders: number (from getRecentConsultationsForDocument)

    2. Export interface MigratorAlert:
       - detected: true
       - severity: "high" | "critical"
       - message: string (human-readable explanation)
       - riskFactors: string[] (list of factors that triggered the alert)
       - cancelledAt: { providerName: string, daysOverdue: number, overdueAmount: number }
       - consultingProvider: { id: number, name: string }
       - recentConsultations: number
       - alertRecord: InsertAntiFraudAlert (ready to insert into DB)

    3. Export function detectMigrator(input: MigratorDetectionInput): MigratorAlert | null
       Detection logic — ALL three conditions must be true:
       a) CPF has a customer record at another provider (not consultingProviderId) with:
          - status containing "cancelado" or "cancelled" or "desativado" (case-insensitive)
          - OR maxDaysOverdue > 90 (implies long-standing debt, likely cancelled)
          - AND the record is "recent" — if registrationDate exists, cancellation must be within last 90 days
       b) CPF has active debt (totalOverdueAmount > 0) at any provider in the region
       c) The consultation is being made by a DIFFERENT provider than where the debt exists

       Severity calculation:
       - "critical" if recentConsultationsByDistinctProviders >= 3 (CPF is "shopping around")
       - "high" otherwise

       If all conditions met, return MigratorAlert with:
       - message: "MIGRADOR SERIAL: CPF com contrato cancelado em {providerName}, divida ativa de R$ {amount}, e {N} consultas recentes por provedores diferentes"
       - riskFactors: ["contrato_cancelado_recente", "divida_ativa", "consulta_outro_provedor", optionally "multiplas_consultas"]
       - alertRecord: InsertAntiFraudAlert with type "migrador_serial", the cancelled provider's providerId, severity, all relevant fields

    4. Export function buildMigratorAlertRecord(alert data): InsertAntiFraudAlert
       - Helper that builds the DB-ready alert record
       - Sets type = "migrador_serial"
       - Sets providerId = the provider where debt exists (the "victim" provider)
       - Sets consultingProviderId and consultingProviderName
       - Sets riskFactors as jsonb array

    IMPORTANT: This function does NOT persist the alert — it returns the alert record for the route to persist. This keeps the service pure and testable.

    IMPORTANT: The status field matching must handle variations: "cancelado", "cancelled", "desativado", "inativo", "desconectado". Use case-insensitive includes check.
