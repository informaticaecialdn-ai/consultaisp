Output: `server/services/address-search.service.ts` with exported functions ready to be wired into the consultas route.

  <action>
    Create server/services/address-search.service.ts with:

    1. Export interface AddressRiskScore:
       - riskScore: number (0-100, 0=safe, 100=highest risk)
       - cpfsDistintosInadimplentes: number
       - totalOcorrenciasEndereco: number
       - riskLevel: "baixo" | "moderado" | "alto" | "critico"
       - alertas: string[]

    2. Export interface AddressSearchResult:
       - cep: string (the searched CEP)
       - logradouro?: string
       - addressGroups: Array of { addressHash: string, cep: string, numero: string, complemento?: string, customers: customer[] with providerName and providerId }
       - totalCustomersFound: number
       - totalProvidersResponded: number
       - risk: AddressRiskScore
       - erpSummary: { total, responded, failed, timedOut }

    3. Export function groupCustomersByAddress(erpResults: RealtimeQueryResult[]):
       - Flatten all customers from all ERP results
       - Group by address hash (CEP + addressNumber + complement via hashAddressForNetwork)
       - Skip customers without CEP or addressNumber (cannot be grouped)
       - Return Map<addressHash, { cep, numero, complemento, customers[] }>

    4. Export function calculateAddressRisk(groupedCustomers: Map result from above):
       - Count distinct CPFs with maxDaysOverdue > 0 across all address groups
       - Count total occurrences of delinquency
       - Risk scoring:
         * 0 distinct delinquent CPFs -> riskScore 0, riskLevel "baixo"
         * 1 delinquent CPF -> riskScore 25, riskLevel "moderado"
         * 2 delinquent CPFs -> riskScore 60, riskLevel "alto"
         * 3+ delinquent CPFs -> riskScore 85, riskLevel "critico"
       - Add alertas strings similar to F5 in isp-score.ts
       - Return AddressRiskScore

    5. Export function buildAddressSearchResult(cep: string, erpResults: RealtimeQueryResult[]): AddressSearchResult
       - Calls groupCustomersByAddress, then calculateAddressRisk
       - Builds the full AddressSearchResult with erpSummary

    IMPORTANT: This service does NOT call queryRegionalErps itself — it receives the results and processes them. The route (Plan 03) handles the ERP call. This keeps the service pure and testable.

    IMPORTANT: Customer data in addressGroups must be LGPD-masked using maskCrossProviderDetail for cross-provider entries.

    Create server/services/address-search.service.test.ts with vitest tests:
    - Use mock RealtimeQueryResult arrays with fake customer data
