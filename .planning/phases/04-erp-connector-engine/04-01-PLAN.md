    **server/erp/resilience.ts** — Per D-11, D-12 from CONTEXT.md (using p-retry instead of cockatiel per constraint):
    - Create a `CircuitBreaker` class with:
      - Constructor takes `{ maxFailures: number, resetTimeMs: number }` (defaults: 5, 30000)
      - States: CLOSED (normal), OPEN (rejecting), HALF_OPEN (testing)
      - `execute<T>(fn: () => Promise<T>): Promise<T>` — if OPEN and resetTime not elapsed, throw CircuitOpenError. If HALF_OPEN, allow one call — success resets to CLOSED, failure goes back to OPEN. CLOSED tracks consecutive failures.
      - `getState(): "closed" | "open" | "half-open"`
      - `reset(): void` — manual reset
    - Create `withResilience<T>(fn: () => Promise<T>, options?: { retries?: number, minTimeout?: number, circuit?: CircuitBreaker }): Promise<T>`:
      - Wraps fn with p-retry (default: 3 retries, 1000ms minTimeout, exponential factor 2)
      - If circuit provided, wraps the retried function inside circuit.execute()
      - On non-retryable errors (4xx HTTP), abort retry immediately using pRetry.AbortError

    **server/erp/rate-limiter.ts** — Per ERP-14:
    - Internal `Map<string, pLimit.Limit>` keyed by `providerId-erpSource`
    - `getProviderLimiter(providerId: number, erpSource: string, concurrency?: number): pLimit.Limit` — returns existing or creates new p-limit instance (default concurrency: 3)
    - This prevents a single provider from overwhelming an ERP API with concurrent requests

    **shared/schema.ts** — Add fields for Hubsoft OAuth per D-03:
    - Add to erpIntegrations table: `clientId: text("client_id")`, `clientSecret: text("client_secret")`, `extraConfig: jsonb("extra_config")`
    - These are nullable (optional) fields — only Hubsoft uses them
    - After schema change, run `npx drizzle-kit push` to apply migration

    IMPORTANT: Do NOT install cockatiel or bottleneck. Use only p-retry and p-limit which are already installed.
